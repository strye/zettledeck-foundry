# Narrative Review Reference

Tier-based review model for research documents generated by research.deep skill.

---

## Overview

After draft generation (Step 9 in Phase 3), route through narrative reviewer agent for writing quality assessment and improvement.

**Goals**:
1. Improve clarity and readability
2. Ensure proper structure and flow
3. Catch mechanical errors (grammar, spelling, punctuation)
4. Validate format-specific requirements
5. Preserve user's voice and technical accuracy

**Non-goals**:
- Changing factual content or findings
- Second-guessing synthesis or conclusions
- Adding new research or sources

## Review Invocation

**Input to reviewer**:
```
Please review this research document for writing quality and clarity.

Format: {findings | six-pager | white-paper}

Document:
{content from draft.md}
```

**Reviewer returns**:
- Reviewed content (with corrections applied)
- Review notes (what was changed and why)

## Three-Tier Review Model

The narrative reviewer uses a three-tier model for feedback:

### Tier 1: Auto-Apply (Mechanical Issues)

These are objective corrections that improve quality without changing meaning:

**Grammar & Mechanics**:
- Subject-verb agreement errors
- Tense consistency
- Pronoun clarity
- Sentence fragments
- Run-on sentences

**Spelling & Punctuation**:
- Typos and misspellings
- Missing or misplaced commas
- Incorrect apostrophes
- Quotation mark errors

**Formatting**:
- Inconsistent heading levels
- Missing list punctuation
- Improper markdown syntax
- Broken wikilinks or URLs

**Clarity Improvements**:
- Remove redundant words
- Simplify complex sentences
- Replace vague terms with specific ones
- Improve parallel structure in lists

**Action**: Reviewer applies these changes directly to content.

### Tier 2: Flag for Decision (Style & Tone)

These are subjective improvements where user preference matters:

**Voice & Tone**:
- Overly casual vs overly formal language
- Active vs passive voice preference
- First person vs third person
- Technical jargon appropriateness

**Structure**:
- Section ordering
- Paragraph length (too long or too short)
- List vs paragraph format
- Heading phrasing

**Emphasis**:
- Bold/italic usage
- Bullet vs numbered lists
- Callout boxes or highlights

**Action**: Reviewer suggests change in review notes, but does NOT apply. User decides during Gate 2 approval.

Example review note:
```
Consider: Paragraph 3 under "Key Findings" is very long (8 sentences).
Suggestion: Split into two paragraphs at "Additionally, the research shows..."
```

### Tier 3: Structural Suggestions (Architecture)

These are significant changes that may affect content:

**Content Organization**:
- Missing sections (e.g., "You have findings but no recommendations")
- Redundant sections
- Content in wrong section
- Missing transitions between major sections

**Scope Issues**:
- Content that doesn't match stated purpose
- Findings not supported by sources
- Unsupported claims

**Format Compliance**:
- Six-pager missing required sections
- White paper missing abstract or references
- Findings document missing sources list

**Action**: Reviewer notes in review feedback, does NOT apply. User decides whether to address.

Example review note:
```
Structural note: Open Questions section appears before Findings.
Amazon six-pager convention is to put open questions at end.
Consider moving to Appendices.
```

## Format-Specific Validation

### Findings Document

**Required sections**:
- Overview
- Key Findings (at least 1 theme)
- Sources

**Optional sections**:
- Contradictions & Gaps
- Open Questions

**Validation**:
- ✅ Every finding cites at least one source
- ✅ Sources list matches source notes reviewed
- ✅ Confidence level is justified

### Six-Pager (Amazon Narrative)

**Required sections**:
- Introduction
- Problem
- Solution / Approach
- Results & Evidence

**Optional sections**:
- Tenets
- Open Questions
- Appendices

**Amazon conventions**:
- No bullet lists in main body (use prose paragraphs)
- Data in appendices, not inline
- Bottom-line-first: lead with conclusion

**Validation**:
- ✅ Introduction states the goal clearly
- ✅ Problem is well-defined
- ✅ Solution is actionable
- ✅ Evidence supports claims

### White Paper

**Required sections**:
- Abstract
- Introduction
- Analysis
- Findings
- References

**Optional sections**:
- Background
- Recommendations
- Open Questions

**Academic conventions**:
- Abstract is standalone summary
- Headings use hierarchical numbering
- Citations are complete and consistent
- Formal tone throughout

**Validation**:
- ✅ Abstract accurately summarizes content
- ✅ Analysis depth matches white paper expectations
- ✅ References use consistent format

## Review Output Format

Reviewer returns two components:

### 1. Reviewed Content

Draft content with Tier 1 corrections applied.

### 2. Review Notes

Structured feedback on Tier 2 and Tier 3 items:

```markdown
## Review Notes

### Applied Corrections (Tier 1)
- Fixed 3 spelling errors
- Corrected subject-verb agreement in paragraph 2
- Simplified 2 overly complex sentences
- Fixed inconsistent heading levels

### Style Suggestions (Tier 2 — Your Decision)
- Consider splitting long paragraph under "Theme 2" (8 sentences)
- Heading "What We Found" could be more specific: "Key Technical Tradeoffs"
- Passive voice in Results section — suggest active voice for clarity

### Structural Notes (Tier 3 — Your Decision)
- Open Questions section appears before Findings — consider moving to end
- No recommendations section — findings suggest actionable next steps
- Source 3 cited in findings but not listed in Sources section
```

## User Approval (Gate 2)

Present reviewed draft to user:

```
Draft reviewed. {N} mechanical corrections applied.

Review notes:
{Summary of Tier 2 and Tier 3 suggestions}

What would you like to do?

1. Approve — Write to repository as-is
2. Edit — Make changes to draft (I'll wait)
3. Discard — Delete and end session
```

**If user chooses "Edit"**:
- User edits `reviewed-draft.md` directly in scratch space
- User can address Tier 2/3 suggestions or ignore them
- When done editing, user confirms: "Done editing"
- Proceed to repository write (Step 11)

---

## Review Agent Requirements

The narrative reviewer agent must support:

**Input parameters**:
- Document content (string)
- Format type (findings | six-pager | white-paper)

**Output format**:
- Reviewed content (string)
- Review notes (structured markdown)

**If agent doesn't support these**, skip review step and warn user.

---

## Error Handling

| Scenario | Behavior |
|----------|----------|
| Narrative reviewer not found | Skip review, warn user, proceed with unreviewed draft |
| Reviewer invocation fails | Log error in session.md, proceed with unreviewed draft, warn user |
| Reviewer returns malformed output | Log warning, use original draft, notify user |
| Review introduces factual errors | User catches during Gate 2, edits draft manually |

## Future Enhancements

Potential improvements to review process:

**Format-specific reviewers**:
- Separate reviewer for Amazon six-pagers (trained on LP writing)
- Academic reviewer for white papers
- Findings reviewer optimized for technical research

**Tiered review levels**:
- Light review (Tier 1 only) — Fast, mechanical corrections
- Standard review (Tier 1 + 2) — Current model
- Deep review (Tier 1 + 2 + 3 with detailed suggestions)

**Review metrics**:
- Readability score (Flesch-Kincaid)
- Sentence complexity analysis
- Passive voice percentage
- Average paragraph length
